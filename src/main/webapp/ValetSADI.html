Overview

A SADI Service Generator for Querying Relational Databases

Valet SADI is a framework that automates the generation of SADI Semantic Web services from declarative service descriptions and semantic mappings to relational data. Mappings are specified in a Datalog sublanguage of Positional-Slotted Object-Applicative (PSOA) RuleML. This page demonstrates a the proof-of-concept implementation and preliminary evaluation of the approach by generating a set of SADI services over a database representing a fragment of a hospital research data warehouse. These automatically generated fully functional SADI services are indexed in a SADI service registry and can then be invoked to answer simple SPARQL queries by the query engines such as HYDRA and SHARE.

Benefits

Automated Code Generation

Ready-to-deploy and executable SADI services are generated from the I/O descriptions.

User friendly

No need to write Java code for the services, especially by the non-technical users.

Query Rewriting

SQL query is generated from the I/O and instantiated from the input RDF data.

Domain Independent

Can be used for any domain if suitable mapping can be written between the ontology and the database.
Framework Components
Generating a complete SADI service relies on four(4) separate modules

Module 1: Semantic Mapping

Represents the correspondence of a database schema to a domain ontology, which must be written manually.

Module 2: Service Description

Service inputs and outputs are formalized using the terminologies from the domian ontology.

Module 3: SQL-template Query Generator

Generates the uninstantiated SQL-template query from I/O descriptions.

Module 4: Service Generator

Generates executable Java code for a SADI web service with the instantiated SQL query.

Separate Ontologies for Domain & Service
  • Domain Ontology
    • Provides a vocabulary for formalizing Semantic Querying of relational data.
    • Focused on a particular domain e.g. Hospital-Acquired Infection (HAI).
    • Domain experts (e.g. infection control practitioners) are familiar with the vocabulary.
  • Service Ontology
    • A separate ontology where the domain expert can define the SADI Service I/O.
    • Uses concepts from the domain ontology and various OWL class constructs.
    • I/O classes define the shape of RDF graph.
Domain Ontology and Service Ontology
Semantic Mapping
  • The mapping is required to capture the underlying database design and provide an explicit semantic correspondence, using rules or some other axioms.
  • Relevant parts of the schema are mapped to the domain ontology using PSOA rules
  • Manually written perhaps by the technical personnel with some knowledge of the database schema.
  • Any changes occurring in the database schema in the future must be reflected in the mappings, to ensure the correct operations of the generated services, and the services must be re-generated.
Mapping Rules to Access Database from Domain Ontology
Service I/O Descriptions
  • Each service is defined using an input OWL class and an output OWL class.
  • Terminologies from the domain ontology are used for definitions.
  • Two SADI services are defined below:

getDiagnosisIDByPatientID

Description: Retrieves diagnosis id (hdgWID) based on the patient id (patWID).
Input: Patient
Output: has_diagnosis some diagnosis

getDiagnosisCodeByDiagnosisID

Description: Retreives diagnosis code (hdgCd) based on the diagnosis id (hdgWID).
Input: Diagnosis
Output: has_diagnosis_code some string

Defining Service I/O based on Domain Ontology Terminologies
How SQL-template Query is generated?
  • SQL-template query is created automatically.
  • Generated based on the service I/O definition.
  • The I/O definition is rewritten into SQL-template query
  • Three inputs are processed by VampirePrime reasoner: I/O definitions, domain ontology, and the mapping rules.
  • Incremental Query Rewriting (IQR) technique together with some auxiliary SQL-specific code is required.
  • All inputs to the VampirePrime reasoner are translated into TPTP formula.
  • The generated query is necessary and sufficient to generate all possible outputs.
Generation of SQL-template Query
Details of the Service Code
  • The Generated Code block is assembled from:
    • An input code block for retrieving the input data value from input RDF graphs and storing the values in local variables,
    • An executable SQL query obtained by substituting the formal parameters in the template with the corresponding variables, and
    • An output code block, consisting of query execution code and some code forming the output RDF graph from the SQL result sets, according to the output class descriptions.
  • The skeleton of the generated service code is organized as a standard Maven Java Web Application.
Complete SADI Service Code Generation
Final Steps
  • The SADI Semantc Web service code is:
    • compiled
    • packaged, and
    • deployed in a special SADI service registry.
  • The deployed SADI services can now be invoked, composed, executed by the HYDRA/SHARE SPARQL engine. They are found based on the properties used in the I/O definitions.
  • The end user executes SPARQL query which contains the RDF input data.
  • The services fetch the data from the database tables. The SQL-template query is instantiated by substituting the formal parameters with specific input values each time the service is executed. The input values are retrieved from the input RDF data.
  • Finally, the end user receives results from HYDRA/SHARE.
